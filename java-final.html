java中的final关键字的思考

最近工作中发现了一个小bug，带来了非常大的影响。是跟java中的final关键字有关。

java中的final关键字这里不做过多的介绍，它可以修饰在数据字段、方法、类，这里主要说final修饰在数据字段上的情况：

1，如果字段是基本数据类型，该字段的值不可改变。

2，如果字段是对象引用，该字段的引用不可改变。

对于第二种情况，字段的引用虽然不可改变，可是对象本身内的其他属性确是可以改变的，因此有些情况下，我们希望一个对象字段永远不要改变，因此使用了final字段，但是其实没有限制住对这个对象的修改，结果造成一些问题。

看下这次的bug，下面是一个共用的返回值DTO（某个开源框架的代码），很多不同的service方法的返回值都是用的这个类

下面是一般的service方法，对于一般成功失败的，直接返回ReturnT类里已经定义好的SUCCESS、FAIL这两个final变量，不需要再重新创建新的ReturnT对象，并赋值。

最近有一段新上线的service代码这样写了：

method2如果发生异常，将ReturnT.SUCCESS这个final对象的code直接改成失败了，那么之后method1方法执行成功返回的ReturnT.SUCCESS，实际上返回的是一个code=FAIL_CODE的对象，并不符合期望。结果造成大面积的像method1这种写法的service，全部返回一样的失败结果。（但是其实业务逻辑是正确执行了的）

对此有两方面的反思：

  1. 一个对象用final修饰了，我们要不要去改对象内的属性内容？
  2. 在java中，当我们希望我们定义的一个字段，引用及对象内的属性都不可改变的时候，怎么做比较好？

对于第一点，我的看法是尽量不要，我们无法猜测，定义这个字段的开发者（使用了final修饰）的明确意图，是不是故意允许其他人可以修改这个变量，有可能就向上面ReturnT这个类的SUCCESS和FAIL字段，开发者其实是不希望有人能修改这个变量的。当然也有其他的场景，比如开发者只是想让引用不可变，对象可以随意修改。这个需要大家自己去仔细评估。

对于第二点，首先对这个对象需要做一些改动，其实effective java第15条中有详细的介绍，需要做到一下几点：

  • 不要提供任何会修改对象状态的方法。（称为mutator）
  • 保证类不会被扩展。
  • 使所有域都是final的。
  • 使所有域都是私有的。
  • 确保对于任何可变组件的互斥访问。
